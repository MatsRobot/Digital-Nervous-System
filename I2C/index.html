🖇️ I2C: Inter-Integrated Circuit

I2C is the "Local Area Network" for the robot's sensory organs. Unlike the point-to-point nature of UART, I2C is a Shared Bus architecture that allows the ESP32 Controller to communicate with multiple peripherals using just two wires.

Core Role: Orchestrating the sensor suite, including IMUs (MPU6050), OLED displays, and Servo drivers (PCA9685).

📟 Protocol Logic: Synchronous Dialogue

I2C is a synchronous protocol. It uses a dedicated Clock (SCL) wire to synchronize data sampling on the Data (SDA) wire. This allows for reliable communication without the baud-rate sensitivity of UART.

✅ Strengths

  • Multi-Device: Connect up to 127 devices on the same 2 wires.
  • Hardware ACK: Built-in confirmation that a device received the data.
  • Simplicity: No complex baud-rate syncing required.

⚠️ Weaknesses

  • Half-Duplex: Devices cannot send and receive at the same time.
  • Address Limits: Some sensors share hardcoded addresses, causing conflicts.
  • Capacitance: Speed drops as more devices/wires are added to the bus.

📦 The I2C Packet Sequence

Communication is always initiated by the Master (ESP32) through a specific bit-handshake:

START
ADDR (7-bit)
R/W Bit
ACK
DATA (8-bit)
ACK
STOP

🧪 Experimental Design: Pull-ups & Scanning

I2C lines are "Open-Drain," meaning they cannot pull the signal HIGH on their own. Our experiment focuses on bus stability:

  1. Pull-up Resistors: We use 4.7kΩ resistors to ensure signals return to 3.3V correctly.
  2. Bus Scanning: The robot runs a startup scan to detect the Hex IDs (e.g., 0x68 for IMU). If a sensor is missing, the AI enters a failsafe state.
  3. Clock Stretching: Configured to allow slower sensors (like the Nano nodes) to pause the Master if they need processing time.
